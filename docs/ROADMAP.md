# Дорожная карта AI Lawyer

## Вдохновение

Помимо юридических систем, мы изучаем лучшие практики из инструментов управления знаниями — в частности [Tana](https://tana.inc/), которая строится на графе связей + AI. Ключевые идеи, которые мы заимствуем:
- **Типизированные узлы (supertags)** — каждый узел в ПГС имеет тип и наследует набор полей
- **Голосовой ввод** — пользователь описывает ситуацию голосом, система структурирует
- **Мульти-LLM** — выбор модели под задачу (Claude для генерации, мелкая модель для классификации)
- **Динамические представления** — одни и те же данные, разные срезы (по типу нарушения, по маркетплейсу, по статье закона)

Разница: Tana — универсальный инструмент для всех. Мы — вертикальное решение для юридических вопросов в e-commerce, где глубина экспертизы важнее широты.

## Фаза 1: Фундамент (2–4 недели) ✅ ВЫПОЛНЕНО

**Цель:** Рабочий MVP — можно задать вопрос, получить ответ со ссылками на документы.

**Что сделано:**
- Собрана база знаний из 8 документов (законы, правила маркетплейсов, шаблоны, FAQ)
- RAG-движок на LlamaIndex + ChromaDB с локальными эмбеддингами (multilingual-e5-base)
- Агент на LangGraph с графом состояний (почемучка → ретривал → генерация → валидация)
- Telegram-бот (aiogram 3) с поддержкой загрузки документов
- FastAPI REST API
- CLI-демо (полный RAG) + автономное демо (TF-IDF, без зависимостей)

**Результат:** Telegram-бот и API, которые отвечают на юридические вопросы со ссылками на конкретные статьи и документы.

## Фаза 2: ПГС и память (3–4 недели)

**Цель:** Система начинает «помнить» и строить связи. ПГС становится «сердцем» системы.

**Что делать:**

Ядро ПГС:
- Поднять Neo4j
- Реализовать связи между документами: «этот закон ссылается на тот», «эта поправка меняет тот пункт»
- Записывать прецеденты: какие вопросы задавали, какие документы помогли, как решили
- Обогащать ПГС после каждого взаимодействия
- Настроить дерево изменений (когда приходит новая версия документа)

Типизация узлов (идея из Tana — supertags):
- Ввести систему типов для узлов ПГС: «Статья закона», «Прецедент», «Маркетплейс», «Тип нарушения», «Шаблон документа»
- Каждый тип наследует набор полей (например, «Статья закона» → номер статьи, закон, дата, связанные статьи)
- Автоматическая типизация при загрузке — система сама определяет тип узла по содержанию
- Динамические запросы: «все статьи, связанные с возвратом товаров» или «все прецеденты по Wildberries за последний месяц»

**Результат:** Система становится умнее с каждым запросом. Типизированный граф позволяет находить неочевидные связи между законами и ситуациями.

## Фаза 3: Валидация и безопасность (2–3 недели)

**Цель:** Минимизация ошибок. Пользователь может доверять ответам.

**Что делать:**
- Усилить `Validator`: проверка номеров статей по базе знаний
- Перекрёстная валидация: второй вызов LLM проверяет первый
- Система уровней уверенности (🟢 / 🟡 / 🟠)
- Аудит-лог всех действий (PostgreSQL)
- Шифрование персональных данных
- Тесты: набор из 50+ юридических вопросов с эталонными ответами

Мульти-LLM стратегия (идея из Tana):
- Claude Opus/Sonnet — для генерации юридических ответов (основная задача)
- Лёгкая модель (Haiku / локальная) — для классификации документов, рутинной валидации
- Возможность переключения модели в зависимости от сложности запроса
- Это снизит стоимость и ускорит обработку простых запросов

**Результат:** Системе можно доверять. Ошибки ловятся до того, как попадут к пользователю. Оптимальное соотношение качество/стоимость через мульти-LLM.

## Фаза 4: Пузыри и специализация (4–6 недель)

**Цель:** Система сама организует знания вокруг популярных тем.

**Что делать:**

Самоорганизация:
- Автоматическое обнаружение пузырей (`detect_bubbles`)
- Кристаллизация пузырей: выделение экосистемы с собственным контекстом (по аналогии с NotebookLM)
- Ротация агентов: автоматическая замена при заполнении контекста
- Специализированные агенты внутри пузырей (со своими промптами и контекстом)

Динамические представления (идея из Tana):
- Одна и та же база знаний — разные «линзы» просмотра:
  - По типу нарушения (некачественный товар, задержка доставки, отказ в возврате)
  - По маркетплейсу (Wildberries, Ozon, Яндекс Маркет)
  - По статье закона (ст. 18 ЗоЗПП, ст. 26.1, ст. 475 ГК РФ)
  - По этапу решения проблемы (претензия → Роспотребнадзор → суд)
- Веб-интерфейс с визуализацией графа и панелью источников
- Пользователь видит не просто ответ, а карту связей вокруг своей проблемы

**Результат:** Полноценная самоорганизующаяся система с эмерджентной специализацией и наглядной навигацией по знаниям.

## Фаза 5: Голос и UX (2–3 недели)

**Цель:** Нулевой порог входа — пользователь просто рассказывает ситуацию.

**Что делать:**

Голосовой ввод (идея из Tana):
- Интеграция speech-to-text (Whisper / Yandex SpeechKit для русского)
- Пользователь записывает голосовое в Telegram → система транскрибирует → извлекает юридическую суть → даёт ответ
- Автоматическое структурирование: из свободного рассказа выделяются факты (что купил, когда, где, что не так)

Генерация документов:
- По результатам разговора система предлагает: «Составить претензию?»
- Автоматическая генерация готовой претензии с заполненными полями
- Экспорт в PDF / DOCX

**Результат:** Человек без юридического образования за 2 минуты голосом описывает ситуацию и получает готовую претензию с правовым обоснованием.

**Цель:** Система распространяется как вирус.

**Что делать:**
- Мультитенантность: изоляция данных между клиентами
- API для интеграций (CRM, бухгалтерия, ERP)
- Сетевой эффект: общие знания обогащают систему для всех (анонимизированные прецеденты)
- Монетизация: freemium, подписка, pay-per-query
- Мобильное приложение
- Поддержка нескольких юрисдикций (Казахстан, Беларусь, Узбекистан)
- Партнёрства с юридическими фирмами: сложные кейсы передаются живым юристам

**Результат:** Продукт, от которого невозможно отказаться.

## Принципы движения между фазами

1. **Не переходи к следующей фазе, пока предыдущая не работает стабильно.** Лучше простой работающий бот, чем сложная сломанная система.

2. **Обратная связь от пользователей — на каждом этапе.** Даже на Фазе 1 дать попробовать людям.

3. **Данные важнее кода.** Качество базы знаний определяет качество ответов.

4. **Каждая фаза должна приносить пользу** сама по себе, а не только как подготовка к следующей.

5. **Учись у лучших, но оставайся в нише.** Tana, NotebookLM, Perplexity — вдохновение. Но наша сила — глубина в юридическом e-commerce, а не широта.
